Acceptance TDD explained

User stories

  • we need to know how our system should behave
  • a simple way of expressing requirements: who does what (and why)
  • Hannah Arendt: "storytelling reveals meaning without committing the error of defining it"

Acceptance tests

  • specifications for the desired behavior and functionality of a system

  • An example user story: Support technician sees customer's history on screen at the start of a call.

    Exaple tests:

    • Simulate a call with Fred's account number and verify that Fred's info can be read from the screen;
    • Verify that the system displays a valid error message for a non-existing account number;
    • Omit the account number in the incoming call completely and verify that the system displays the text "no account number provided" on the screen.

  • owned by the customer
  • written together with the customer, developer and tester
  • expressed in the language of the problem domain
  • concise, precise and unambiguous
  • focus on the what, not the how
  • a test with too much detail: go to the new transaction screen, fill in the required details and save the entry; verify that the transaction shows up in the list

Understanding the process

  • a cycle consisting of: pick a story, write tests, automate tests, implement functionality
